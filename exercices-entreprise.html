Exercices Entreprise

4 cas professionnels — max 6 tables

4 cas concrets, contexte professionnel

Chaque cas couvre DD → MCD → MLD → SQL. Inspirés de vrais besoins d'entreprise.

Cas 1 — VeloShop : Gestion commerciale de velos

5 tables
VeloShop est un magasin de velos a Lausanne. Il vend des articles (velos, accessoires) fournis par differents fournisseurs. Les clients passent des commandes contenant plusieurs articles avec des quantites. Chaque article appartient a une famille (VTT, Route, Urbain, Accessoire).
FAMILLE(#nofam, libfam)
FOURNISSEUR(#nofour, nomfour, villefour)
ARTICLE(#noart, desart, prix, qtesto, →nofam, →nofour)
CLIENT(#nocli, nomcli, prenomcli, villecli, cacli)
COMMANDE(#nocde, datecde, →nocli)
LIGNECDE(→#nocde, →#noart, qtecde)
P1 Conception
  1. Identifier les entites et attributs a partir du contexte
  2. Dessiner le MCD avec cardinalites Merise
  3. Identifier les CIF et transformer en MLD
P2M2 Developpement
  1. Ecrire le script CREATE TABLE (ordre FK correct)
  2. Inserer 10 articles, 5 clients, 8 commandes avec lignes
  3. Requetes : articles en rupture (qtesto = 0), CA par client, commandes du mois
  4. JOIN : commandes avec nom client + detail articles
M3D2 Analyse
  1. Dashboard : top 5 articles vendus, CA total, nb commandes par mois
  2. Evaluer : quelles ameliorations pour VeloShop ? (stock auto, fidelite client...)

Cas 2 — TechRepair : Gestion d'interventions

5 tables
TechRepair est une societe de maintenance informatique a Geneve. Des techniciens interviennent chez des clients pour reparer du materiel. Chaque intervention est liee a un client, un technicien et un type de panne. On enregistre la duree et le statut (en cours, terminee, facturee).
TECHNICIEN(#id_tech, nom, prenom, specialite, tarif_h)
CLIENT(#id_client, raison_sociale, adresse, telephone, email)
TYPE_PANNE(#id_panne, libelle, priorite)
INTERVENTION(#id_interv, date_interv, duree_h, statut, →id_tech, →id_client, →id_panne)
FACTURE_INTERV(#id_fact, date_fact, montant, statut_paie, →id_interv)
P1 Conception
  1. Dictionnaire des donnees (identifier les calculees : montant = duree × tarif)
  2. MCD avec cardinalites + CIF
  3. MLD textuel + verifier 3NF
P2P3 Developpement
  1. CREATE TABLE + contraintes (CHECK duree > 0, ENUM statut)
  2. INSERT donnees realistes (8 techniciens, 15 clients, 30 interventions)
  3. Requetes : interventions par technicien, pannes les plus frequentes, temps moyen par type
  4. JOIN : interventions avec nom technicien + nom client + type panne
M3 Analyse
  1. KPI : taux de resolution, CA par technicien, repartition des pannes

Cas 3 — MeetingHub : Reservation de salles

4 tables
MeetingHub gere les salles de reunion d'un immeuble de bureaux a Berne. Les employes reservent des salles pour des creneaux horaires. Chaque salle a une capacite et un equipement (projecteur, visio, tableau blanc). Chaque employe appartient a un departement.
DEPARTEMENT(#id_dept, nom_dept)
EMPLOYE(#id_emp, nom, prenom, email, →id_dept)
SALLE(#id_salle, nom_salle, capacite, equipement, etage)
RESERVATION(#id_resa, date_resa, heure_debut, heure_fin, objet, →id_emp, →id_salle)
P1M1 Conception + normalisation
  1. MCD complet (attention : verifier qu'une salle ne peut pas etre reservee 2 fois au meme creneau)
  2. MLD + MPD avec types MySQL et CHECK (heure_fin > heure_debut)
  3. Verifier 3NF : est-ce que nom_dept doit etre dans EMPLOYE ?
P2P3 Developpement
  1. Script complet + peuplement
  2. Requete : salles disponibles a une date/heure donnee
  3. Taux d'occupation par salle (nb heures reservees / nb heures ouvrables)

Cas 4 — HelpDesk : Gestion de tickets support

6 tables
HelpDesk est le systeme de support IT d'une entreprise a Zurich. Les utilisateurs ouvrent des tickets classes par categorie (reseau, logiciel, materiel, acces). Chaque ticket est assigne a un agent du support. Les agents ont un niveau (N1, N2, N3). On suit l'historique des actions sur chaque ticket (commentaires, changements de statut).
CATEGORIE(#id_cat, nom_cat, description)
AGENT(#id_agent, nom, prenom, email, niveau)
UTILISATEUR(#id_user, nom, prenom, email, departement)
TICKET(#id_ticket, titre, description, priorite, statut, date_creation, date_resolution, →id_user, →id_agent, →id_cat)
ACTION_TICKET(#id_action, date_action, type_action, commentaire, →id_ticket, →id_agent)
SLA(#id_sla, →id_cat, priorite, delai_max_h)
P1M1D1 Conception complete
  1. DD complet + matrice des DF
  2. MCD avec 6 entites + cardinalites + CIF
  3. MLD + normalisation 3NF
  4. MPD avec types, contraintes, ENUM pour statut/priorite/niveau
  5. D1 : evaluer le design (forces/faiblesses/ameliorations)
P2P3M2M3D2 Developpement complet
  1. Script DDL + DML + 20 tickets avec actions
  2. Vue : tickets ouverts avec info agent + utilisateur + categorie
  3. Procedure : assigner_ticket(id_ticket, id_agent)
  4. Requetes : temps moyen de resolution par categorie, tickets depassant le SLA
  5. Securite : 3 roles (admin, agent, utilisateur) avec droits differencies
  6. D2 : evaluer l'efficacite + proposer des ameliorations
